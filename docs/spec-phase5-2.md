## 1. 概要

- 何を作るか: 本棚への取り込み導線の拡張と、テキストの表現力向上
- なぜ作るか: 運用で判明した「一括登録できない」「画像/PDFが本棚に入らない」「txtでは挿絵やフォント指定ができない」という制約を解消する
- 誰が使うか: ショウゴさん（Pixel 10 / フランのブラウザ）

## 2. 前提

### 2.1 現行データ構造（変更箇所のみ記載）

```typescript
// 既存BookEntry — fileType を拡張する
type BookEntry = {
  id: string;
  title: string;
  author: string;
  fileType: 'mkb' | 'md' | 'txt' | 'html' | 'json' | 'images' | 'pdf'; // ← pdf 追加
  fileData: ArrayBuffer;
  addedAt: number;
  lastOpenedAt: number;
  tags?: string[];        // ← 新規追加。タグによる分類
  localSettings?: LocalSettings;

```

### 2.2 既存資産の確認

| 項目 | 状態 |
|---|---|
| Markdownレンダリング | react-markdown + remark-gfm + rehype-raw で実装済み。変更不要 |
| 画像表示 | ImageViewer.jsx で実装済み。変更不要 |
| PDF表示 | 未対応。新規追加 |
| 本棚保存（useBookshelf.js） | saveBook() インターフェース安定。tags フィールド追加のみ |

### 2.3 設計判断

**D-008: 一括登録は本棚画面のアクションとする**
ビューア画面の「本棚に保存」ボタン（1件ずつ）は既存のまま残す。一括登録は本棚画面に「まとめて追加」ボタンを新設し、ファイル選択→一括処理のフローを取る。理由: ビューアを経由せずに登録できる方が、大量取り込み時の操作回数が少ない。

**D-009: txt→md変換は読み込み時ダイアログで行う**
txtファイルを読み込んだ時点で変換ダイアログを表示する（縦書き変換ボタンと同じタイミング）。ビューアヘッダーには追加しない。ユーザーが「Markdownに変換」か「テキストのまま」を選択する。変換後のfileTypeは `'md'` として保存し、元のtxtエントリは残さない。自動変換のルールは以下の通り:
- 空行で区切られた最初の行を `# 見出し` に変換
- 連続する空行を段落区切りとして保持
- 既にMarkdown記法（#, *, -, ```, > 等）を含む場合はそのまま保持
- 変換結果をプレビュー表示し、ユーザーが確認してから登録する

**D-011: txtエンコーディングのフォールバック**
txtファイル読み込み時のデコード順序: UTF-8 → Shift_JIS（CP932）。UTF-8でデコードして文字化けを検出した場合、Shift_JISで再デコードを試みる。それでも文字化けする場合はエラーを表示し、取り込まない。文字化け検出は replacement character（U+FFFD）の出現で判定する。

**D-010: PDF表示はiframeで行う**
PDFビューアをゼロから実装するコストは不釣り合い。ブラウザ内蔵のPDFレンダラに任せ、`<iframe src="blob:..." />` で表示する。HTMLビューアと同じスクロール固定方式。

## 3. 技術選定

| 技術 | 用途 | 理由 |
|---|---|---|
| 既存 react-markdown | Markdown描画 | 変更不要。すでに画像埋め込み・見出し・強調に対応 |
| iframe + blob URL | PDF表示 | 外部ライブラリ不要。ブラウザ内蔵機能で十分 |
| TextDecoder('shift_jis') | Shift_JISデコード | ブラウザ標準API。Chrome/Firefox/Edge対応 |
| File System Access API (フォールバック: input webkitdirectory) | フォルダ選択 | フォルダ構造ごとの一括取り込みに必要 |

## 4. 機能仕様

### §31: 複数ファイル一括本棚登録

**何を:** 本棚画面から複数ファイルを選択し、一度の操作で全て本棚に登録する。フォルダ選択時は親フォルダ名をタグとして自動付与する。

**どのように:**

1. Bookshelf.jsx に「まとめて追加」ボタンを新設する
2. タップすると選択モードUIを表示する:
   - 「ファイルを選択」ボタン — `<input type="file" multiple>` で複数ファイル選択
   - 「フォルダを選択」ボタン — `<input type="file" webkitdirectory>` でフォルダ一括選択
3. ファイル選択後、取り込み対象の一覧をプレビュー表示する:
   - ファイル名、推定fileType、サイズ
   - フォルダ選択の場合、各ファイルの親フォルダ名をタグとして表示
   - 対応外のファイルはグレーアウトし「対応外」と表示。取り込み対象から除外する
   - txtファイルには「MDに変換」トグルを表示（デフォルトOFF）
4. 「すべて登録」ボタンで一括保存する:
   - 各ファイルに対してuseMkbLoader相当のパース処理を行い、BookEntryを生成する
   - 進捗バー（N / M ファイル処理中）を表示する
   - 同名ファイルが既に本棚にある場合はスキップし、完了後にスキップ一覧を表示する
5. 完了後、本棚一覧に遷移する

**BookEntryへのtags追加:**

- useBookshelf.js の saveBook() は既にBookEntry型を受け取るため、tags フィールドが追加されるだけで既存インターフェースは維持される
- フォルダ選択時: 各ファイルの直近の親フォルダ名を `tags: ['フォルダ名']` として設定する
- ファイル選択時: `tags: []`（空配列）
- §27のタグ機能（既存）でフィルタリング可能

**対応ファイル形式:**

既存の対応形式をすべて受け付ける: `.mkb`, `.md`, `.txt`, `.html`, `.json`, `.cbz`, `.zip`, `.jpg`, `.jpeg`, `.png`, `.gif`, `.webp`, `.pdf`

**制約:**
- webkitdirectory は Chrome/Edge/Firefox で対応。Safari iOS は非対応だが、Pixel 10 + Chrome が主要ターゲットなので許容する
- フォルダ選択時、ネストしたサブフォルダは直近の親のみタグにする（最上位フォルダ名ではない）
- 1回の一括登録は最大100ファイルとする。超過時は「100ファイルずつ登録してください」と案内する

### §32: 画像・PDF単体の本棚登録

**何を:** 画像ファイルとPDFを、独立したエントリとして本棚に保存できるようにする。

**どのように:**

**画像の本棚登録:**
- 現状、画像ファイルを開くとImageViewer.jsxで表示できるが、本棚保存のフローが未接続
- App.jsx のビューアヘッダーにある「本棚に保存」ボタンのロジックを修正し、ViewerContent.type が `'images'` の場合もsaveBook()を呼べるようにする
- fileType は既存の `'images'` を使用する
- title はファイル名（zipの場合はzip名、単体画像はファイル名）

**PDF対応（新規）:**
- useMkbLoader.js のファイルタイプ判定に `.pdf` を追加する
- ファイル選択の accept に `.pdf` を追加する
- ViewerContent に `{ type: 'pdf'; data: ArrayBuffer }` を追加する
- PdfRenderer.jsx を新規作成する:
  - ArrayBuffer → Blob → blob URL を生成
  - `<iframe src={blobUrl} style="width:100%; height:100vh; border:none" />` で表示
  - ブラウザ内蔵PDFビューアに描画を委ねる
  - テーマ色の注入は不要（PDF自体のスタイルが使われる）
  - ページネーションはブラウザ内蔵ビューアに任せる
- BookEntry の fileType に `'pdf'` を追加する（§2.1参照）
- 本棚保存フローは画像と同様、ビューアヘッダーの「本棚に保存」ボタンで保存する

**制約:**
- PDFのテキスト検索・注釈機能は実装しない（ブラウザ内蔵に任せる）
- 大容量PDF（50MB超）は登録時に警告を表示する。IndexedDBの容量を圧迫するため

### §33: txt→Markdown変換

**何を:** テキストファイルを読み込む際に、Markdownに変換して登録するオプションを提供する。

**どのように:**

**変換ロジック（utils/txtToMd.js 新規作成）:**

```
入力: プレーンテキスト文字列
出力: Markdown文字列

1. 先頭の空行をスキップする
2. 最初の非空行を `# タイトル` に変換する（ただし既に # で始まる場合はそのまま）
3. 以降の行に対して:
   - 既にMarkdown記法を含む行（#, *, -, ```, > で始まる行）はそのまま保持
   - 空行はそのまま保持（段落区切り）
   - 一般テキスト行はそのまま保持（Markdownでは改行が無視されるため、行末に2スペースを付与する）
4. 変換結果を返す
```

**UIフロー（単体ファイル）:**
- txtファイルを読み込んだ時点で変換ダイアログを表示する（縦書き変換ボタンと同じタイミング・同じ導線）
- ダイアログ内容:
  - 変換プレビュー（変換前 / 変換後をタブまたはスワイプで切り替え）
  - 「Markdownに変換」ボタン → txtToMd.js で変換し、fileType `'md'` としてビューアに渡す
  - 「テキストのまま」ボタン → fileType `'txt'` のままビューアに渡す
- 変換後に本棚に保存した場合、fileType は `'md'` として保存される。元のtxtエントリは作らない

**エンコーディング処理（utils/txtToMd.js に含める）:**
- デコード順序: UTF-8 → Shift_JIS（CP932）
- UTF-8でデコードし、replacement character（U+FFFD）が含まれる場合、Shift_JISで再デコードする
- Shift_JISデコードには TextDecoder('shift_jis') を使用する
- Shift_JISでもU+FFFDが含まれる場合、エラーダイアログを表示して取り込みを中止する
- エラーメッセージ: 「このファイルのエンコーディングを判別できませんでした。UTF-8またはShift_JISに変換してから取り込んでください」

**UIフロー（一括登録時）:**
- §31の一括登録プレビュー画面で、txtファイルに対して「MDに変換」トグルを表示する
- ONにすると、そのファイルは変換処理を経て `fileType: 'md'` として登録される
- OFFの場合は `fileType: 'txt'` のまま登録される

**制約:**
- 変換は推定に基づくため、完璧な結果は保証しない。ユーザーがプレビューで確認する前提
- 変換後の手動編集機能は本Phaseでは実装しない（将来の層B編集機能で対応）
- 対応エンコーディングはUTF-8とShift_JIS（CP932）の2種。それ以外（EUC-JP等）は非対応

## 6. テスト方針

| テスト対象 | 方法 | 合格条件 |
|---|---|---|
| 一括登録（ファイル） | 手動: 5つのmd/txtを複数選択して登録 | 全ファイルが本棚に追加される |
| 一括登録（フォルダ） | 手動: フォルダを選択して登録 | 親フォルダ名がタグとして付与される |
| 同名ファイルスキップ | 手動: 既存エントリと同名ファイルを一括登録 | スキップされ、完了後に通知される |
| 画像の本棚保存 | 手動: jpg/pngを開き「本棚に保存」 | 本棚に登録され、再度開ける |
| PDF表示 | 手動: PDFを開く | iframe内でPDFが表示される |
| PDFの本棚保存 | 手動: PDFを開き「本棚に保存」 | 本棚に登録され、再度開ける |
| txt→md変換（単体） | 手動: txtを読み込み、変換ダイアログで「Markdownに変換」 | md として表示・保存される |
| txt→md変換（一括） | 手動: 一括登録でtxtのMD変換をON | mdとして保存される |
| txtエンコーディング（UTF-8） | 手動: UTF-8のtxtを読み込む | 正常にデコードされる |
| txtエンコーディング（Shift_JIS） | 手動: Shift_JISのtxtを読み込む | フォールバックで正常にデコードされる |
| txtエンコーディング（非対応） | 手動: EUC-JPのtxtを読み込む | エラー表示され取り込まれない |
| 既存機能の回帰 | 手動: mkb/md/txt/html/json/images各形式 | Phase 4までの全機能が正常動作 |

## 7. 実装順序の推奨

1. **§32 PDF対応** — ファイルタイプ追加の骨格を作る。影響範囲が小さく、独立して完成する
2. **§32 画像の本棚保存** — App.jsxの保存フロー修正。§32を先にやることで、一括登録の対象形式が確定する
3. **§33 txt→md変換** — エンコーディング判定・変換ロジック・読み込み時ダイアログ。§31の一括登録で「MD変換トグル」を使うため、先に完成させる
4. **§31 一括登録** — 最も影響範囲が広い。§32・§33が完了していれば、全形式を一括で扱える
