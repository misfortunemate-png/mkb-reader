## 0. 設計思想（PGは必ず読むこと）

### ビューアの哲学

このアプリは「ただのビューア」ではない。「徹底的に読み替えられるビューア」である。

原本は一切変更しない。ファイルのバイナリはIndexedDBに保存されたまま、指一本触れない。しかし表示する時に、ユーザーは何でも読み替えることができる。話者名を変え、テキストを差し替え、行を隠し、画像を差し込む。これらは全て「どう読むか」の問題であり、原本への介入ではない。

技術的にはLightroomの非破壊RAW現像と同じモデル。原本 + サイドカー（localSettings）→ 表示。エクスポート時にのみ読み替えを適用した新しいファイルを生成する。

### なぜ「編集」ではなく「読み替え」か

生成AIとのチャットログを考える。アシスタントが返した応答は、サービスが生成したものであり、ユーザーが「編集」する対象ではない。しかしローカルに落としたファイルは個人のものであり、自由に「読み替え」ることができる。話者名を「human」から自分の名前に、「assistant」から家族の名前に。特定の行を非表示にする。テキストの一部を別の表現に差し替える。

この区別は単なる言葉の問題ではなく、アーキテクチャに影響する。「編集」ならundo/redo履歴、差分管理、競合解決が必要。「読み替え」ならルールの追加・削除だけ。実装が一桁軽くなる。

将来、別のプロダクト（プロダクトB）と連携した場合、そちらでは原本そのものを徹底的に編集できるようにする。ビューアとエディタの責務は完全に分離する。

### 設定UIの原則

「正解がないから設定で制御する」。ただし細かい数値を並べるのではなく、プリセットで選ばせる。

- 各設定セクションの主UIはプリセット選択（2〜4択のボタン）
- スライダー・数値入力は「詳細設定」として折りたたむ。普段は見えない
- 設定はグローバル（全ファイル共通）とローカル（ファイルごと）の二層
- ローカル設定が存在すればそちらを優先し、なければグローバルにフォールバック

### このフェーズで実現すること

Claude.aiのチャットログをmkbに変換し、話者名を読み替え、画像を差し込み、読み替え済みの姿をMKBとしてエクスポートする。一連の流れが繋がる。

## 4. 機能仕様

### §18: チャットログ変換（Claude.ai JSON → mkb）

**何を:** Claude.aiからエクスポートしたconversations.jsonを読み込み、各会話をmkb形式に変換してビューアで閲覧・本棚保存できるようにする。

**入力フォーマット:**

Claude.ai Settings → Privacy → Export Data で取得するZIPに含まれる `conversations.json`:
```json
[
  {
    "uuid": "a1b2c3d4-...",
    "name": "会話タイトル",
    "created_at": "2025-11-04T09:22:11.000Z",
    "updated_at": "2025-11-04T10:45:03.000Z",
    "model": "claude-3-5-sonnet-20241022",
    "chat_messages": [
      {
        "uuid": "m1m2m3...",
        "sender": "human",
        "text": "メッセージ本文",
        "created_at": "2025-11-04T09:22:11.000Z"
      },
      {
        "uuid": "m4m5m6...",
        "sender": "assistant",
        "text": "応答本文（Markdown形式）",
        "created_at": "2025-11-04T09:22:14.000Z"
      }
    ]
  }
]
```

**ツリー構造の取り扱い（追補 v1.2）:**

Claude.ai の会話はツリー構造で、メッセージを編集または再生成すると新しい枝が作られ、
古い枝も `chat_messages[]` に残る。会話オブジェクトの `current_leaf_message_uuid`
（フィールド名揺れ: `current_leaf` / `leaf_message_uuid` も許容）が「現在表示中の葉」
を指す。各メッセージの `parent_message_uuid` で親をたどる。

chatConverter.js は次の戦略で「現在表示されている分岐」だけを抽出する:

1. `current_leaf_message_uuid` が指定されていればそれを葉として採用
2. 無ければ「他から親として参照されていない＝葉である」もののうち最後を葉と推定
3. `parent_message_uuid` を辿りながらルートに向かって path を回収
4. ループや欠損を検出した場合は安全のため全件返す（情報を失わないことを優先）

ChatImporter UI には「最終分岐のみ / 全分岐」のトグルを置き、既定は「最終分岐のみ」。
全分岐を残したい場合（編集履歴を比較したい等）はトグルで切替可能。

**メッセージ本文のフォーマット（追補 v1.1）:**

Claude.ai の新エクスポート形式では `chat_messages[].content` が配列で、各要素に `type` がある:
- `type: 'text'` — 通常応答
- `type: 'thinking'` — 拡張思考の独白（応答本文とは別）
- `type: 'tool_use' | 'tool_result'` — ツール呼び出しと結果

これらは「鑑賞」の主役（応答本文）と副次（思考・ツール）を分けるべき情報。
chatConverter.js は次のルールで MD を組み立てる:

- `text` ブロック → そのまま MD 本文として連結
- `thinking` ブロック → `<details class="claude-thinking"><summary>思考</summary>…</details>` で折りたたみ
- `tool_use` ブロック → `<details class="claude-tool"><summary>tool_use: name</summary>…</details>` で折りたたみ
- `tool_result` ブロック → 同上

CSS で控えめな枠と小さめのフォント。既定では折りたたみ、タップで展開。
読み手は応答本文を主に読み、思考・ツールを必要時に開く。

**変換ルール:**

chatConverter.js が行う変換:

1会話 = 1つのMkbData:
- metadata.title = conversation.name（「Untitled」ならuuidの先頭8文字に）
- metadata.author = 空文字（ユーザーが後からローカル設定で読み替える）
- chapters[0] にindex.md（会話全体を1チャプターとして格納）

メッセージのMarkdown変換:
```markdown
**human**

メッセージ本文

---

**assistant**

応答本文（Markdown形式）

---

**human**

次のメッセージ

---
```

各メッセージは `**話者名**` + 空行 + 本文 + `---`（区切り線）で連結する。

設計判断:
- 話者名をMD内にプレーンテキストとして埋め込む理由: 読み替えルール（§14）でテキスト置換の対象にできるため。特別なメタデータ形式にすると置換が複雑になる
- `---`で区切る理由: §7の表示方式設定（改ページ/線/余白/装飾）がそのまま効く。メッセージ間の視覚的区切りをユーザーが制御できる
- 1会話1チャプターの理由: チャプター分割は後でローカル設定やプロダクトBで対応できる。最初はシンプルに

**変換UIフロー（ChatImporter.jsx）:**

1. ファイル選択ボタン（`.json`対応）で `conversations.json` を開く
2. JSON内の全会話をリスト表示する（タイトル / 日付 / メッセージ数）
3. ユーザーが変換したい会話を選択する（複数選択可、全選択ボタンあり）
4. 「変換」ボタンをタップ
5. 選択された各会話をMkbDataに変換
6. 変換結果をビューアで表示する（1つ選択時）/ 本棚に一括保存する（複数選択時）

**アクセス方法:**
- 本棚画面の「＋ 開く」ボタンの隣に「チャットログ取り込み」ボタン（またはメニュー内）
- 最初のファイル選択後にChatImporter画面に遷移

**制約:**
- Claude.aiのエクスポートJSON形式のみ対応（ChatGPT等は将来）
- 会話内の添付ファイル（画像等）はJSONに含まれないため、テキストのみ変換
- 非常に長い会話（数百メッセージ）の場合、変換に数秒かかる可能性がある

### §14: 読み替え設定

**何を:** 本棚内のファイルに対して、表示時にテキストを読み替えるルールを設定・適用する。

**読み替えルールの種類:**

```typescript
type RewriteRules = {
  // 話者名の読み替え
  speakerNames?: {
    human?: string;       // 例: 'ショウゴ'
    assistant?: string;   // 例: 'クリーデ'
  };
  // テキスト置換（表示時に適用。原本は変更しない）
  replacements?: {
    id: string;           // crypto.randomUUID()
    pattern: string;      // 検索文字列
    display: string;      // 表示する文字列
    scope: string;        // 'all' | chapterId
    enabled: boolean;     // 個別に有効/無効を切替可能
  }[];
  // 行の非表示
  hiddenRanges?: {
    id: string;
    chapterId: string;
    startLine: number;
    endLine: number;
    enabled: boolean;
  }[];
};
```

**rewriteEngine.js:**

読み替えの適用は描画の直前で行う。原本のMDテキストを受け取り、読み替えルールを適用した表示用テキストを返す純粋関数。

```typescript
applyRewrite(originalMd: string, rules: RewriteRules, chapterId: string): string
```

適用順序:
1. speakerNames: `**human**` → `**ショウゴ**` のようにBold話者名を置換
2. replacements: enabledなものをscope順に適用
3. hiddenRanges: enabledな範囲の行を除去

speakerNamesの置換は、テキスト全体から `**human**` のパターンを探して置換する。Bold記法の中の話者名だけを対象とし、本文中の「human」は置換しない。

**RewritePanel.jsx:**

ビューア画面のヘッダーに「読み替え」ボタン（ペンアイコン）を追加。タップで読み替え設定パネルを開く。

パネル構成（ボトムシートまたはドロワー）:

```
┌─────────────────────────────────────┐
│ 読み替え設定              [この本]   │
├─────────────────────────────────────┤
│                                     │
│ 話者名                              │
│ human → [ショウゴ        ]          │
│ assistant → [クリーデ      ]        │
│                                     │
│ ─────────────────────────────────── │
│ テキスト置換                    [＋] │
│ ☑ Claude → クリーデ                 │
│ ☑ user → マスター                   │
│                                     │
│ ─────────────────────────────────── │
│ 非表示範囲                      [＋] │
│ （なし）                            │
│                                     │
└─────────────────────────────────────┘
```

- 話者名: テキスト入力欄。入力するとリアルタイムでビューアに反映
- テキスト置換: 「＋」で追加。パターンと表示テキストを入力。チェックボックスで有効/無効
- 非表示範囲: 「＋」で追加。ビューア上でテキストを長押し選択して範囲指定する方式は複雑なので、Phase 3bでは行番号で指定する。将来的にUIを改善

**useRewrite.js:**
- 現在のBookEntryからlocalSettings.rewriteを読み取る
- ルールの追加・変更・削除をuseBookshelf経由でIndexedDBに保存
- ルール変更時にビューアの再描画をトリガー

**MarkdownRenderer.jsxの変更:**
- レンダリング前にrewriteEngine.applyRewrite()を呼ぶ
- 置換されたテキストにはCSSクラス（`rewritten`）を付与し、薄い背景色で読み替え箇所を視覚的に示す（設定でオフにもできる）

**制約:**
- 正規表現は使わない。文字列の完全一致のみ（正規表現はユーザーに難しい）
- 読み替えはあくまで表示時の処理。検索機能（将来実装）は原本テキストを対象とする
- 非表示範囲の行番号は原本基準。読み替え適用後の行番号ではない

### §15: 画像差し込み

**何を:** 本棚内のファイルに対して、表示時に画像を差し込む。原本のMDは変更しない。

**データ構造:**

```typescript
type InsertedAsset = {
  id: string;             // crypto.randomUUID()
  path: string;           // 'assets/inserted-{id}.{ext}'
  data: ArrayBuffer;      // 画像バイナリ（§12のリサイズ適用済み）
  mimeType: string;       // 'image/png' | 'image/jpeg' | etc.
  insertAfter: {
    chapterId: string;
    lineNumber: number;    // この行の後に挿入
  };
  altText: string;         // MD記法の代替テキスト
  enabled: boolean;
};
```

**ImageInserter.jsx:**

画像差し込みのフロー:
1. ビューア画面で「画像を差し込む」操作を開始する（RewritePanel内のボタン、または長押しメニュー）
2. ファイルピッカーで画像を選択
3. §12のリサイズを適用（長辺2048px制限）
4. 挿入位置を指定する（Phase 3bでは「現在表示中のページの末尾」をデフォルトとし、RewritePanel内で行番号を調整可能にする）
5. localSettings.rewrite.insertedAssetsに保存
6. ビューアが再描画され、指定位置に画像が表示される

**rewriteEngine.jsへの追加:**

applyRewrite関数に画像差し込みの処理を追加:
- insertedAssetsを行番号順にソート
- 各挿入位置の行の後に `![altText](blob:url)` のMD記法を挿入
- 挿入された画像はBlob URLに変換して表示

適用順序の更新:
1. speakerNames
2. replacements
3. hiddenRanges
4. **insertedAssets**（最後に適用。非表示範囲の中に画像を挿入しないようにするため）

**制約:**
- 差し込める画像形式: JPG, PNG, GIF, WebP
- 差し込み位置は行番号指定。将来ドラッグ&ドロップで位置調整できるようにしたい
- 差し込んだ画像はIndexedDBに保存される（localSettings内）。ファイルサイズに注意

### §16: MKBエクスポート

**何を:** 読み替えルールを適用した結果を、新しいMKBファイルとしてエクスポートする。

**エクスポートフロー:**

1. ビューア画面のヘッダーに「エクスポート」ボタン（↓アイコン）を追加
2. タップ → ExportDialog.jsx を表示

```
┌─────────────────────────────────────┐
│ MKBエクスポート                     │
├─────────────────────────────────────┤
│                                     │
│ タイトル: [創作基盤構想書     ]      │
│ 著者:     [ショウゴ           ]      │
│                                     │
│ ☑ 読み替えを適用する                │
│                                     │
│ [エクスポート]                       │
│                                     │
└─────────────────────────────────────┘
```

3. 「エクスポート」タップ

**useExport.js:**

```typescript
exportMkb(bookEntry: BookEntry, options: ExportOptions): Promise<Blob>
```

処理:
1. 原本を展開する（mkbならZIP展開、mdならそのまま）
2. 読み替えルールを適用する（rewriteEngine.applyRewrite()）
3. insertedAssetsの画像をassets/に追加する
4. markbook.yamlを生成する（タイトル、著者、ページ順序）
5. JSZipで.mkbファイルを構築する
6. ブラウザのダウンロードAPIでファイルを保存する

読み替え適用時の注意:
- 「読み替えを適用する」がオフの場合、原本そのままをエクスポート
- hiddenRangesは行を完全に削除する（読み替え済みの原稿として）
- insertedAssetsの画像はassets/に配置し、MD内の参照はファイルパスに変換（Blob URLではなく`assets/inserted-xxx.png`形式）

**制約:**
- エクスポートされたMKBは「新しい原本」。読み替えルールは含まれない
- エクスポート後も元のファイルと読み替えルールはそのまま残る
- 非常に多くの画像を含む場合、ZIP構築に数秒かかる。プログレス表示を入れる

## 5. テスト方針

| テスト対象 | 方法 | 合格条件 |
|---|---|---|
| §18: JSON読み込み | 手動。Claude.aiエクスポートJSON（テスト用）を開く | 会話リストが表示される |
| §18: 変換 | 手動。会話を1つ選択して変換 | ビューアで表示され、human/assistantの発言が区切られている |
| §18: 一括保存 | 手動。複数会話を選択して本棚に保存 | 全てが本棚に追加される |
| §14: 話者名読み替え | Pixel 10実機 | human→ショウゴ、assistant→クリーデに表示が変わる |
| §14: テキスト置換 | Pixel 10実機 | 指定した文字列が置換されて表示される |
| §14: 置換ハイライト | Pixel 10実機 | 読み替え箇所が薄い背景色で示される |
| §14: 非表示範囲 | Pixel 10実機 | 指定行が表示されない |
| §14: ルール永続化 | Pixel 10実機 | アプリ再起動後も読み替えルールが維持される |
| §15: 画像差し込み | Pixel 10実機 | ファイルピッカーで画像を選び、指定位置に表示される |
| §15: リサイズ適用 | 手動。3000px画像を差し込む | 2048px以下にリサイズされて保存される |
| §16: エクスポート | Pixel 10実機 | .mkbファイルがダウンロードされる |
| §16: 読み替え適用エクスポート | 手動。読み替えありでエクスポート | エクスポートしたmkbを再度開くと読み替え済みのテキストが表示される |
| §16: 読み替え非適用エクスポート | 手動 | 原本のままエクスポートされる |
| 統合: 一連の流れ | Pixel 10実機 | JSON取り込み→話者名読み替え→画像差し込み→MKBエクスポートが一連で動作する |
